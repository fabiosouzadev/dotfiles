<summary>
Achados acionáveis para adicionar um guard declarativo `block_on_workplace` no script
`run_once_after_510-install-ai-tools.sh.tmpl`, respeitado genericamente nos dois ranges

<findings>

## 1. Escopo `.` vs `$` dentro do range — CONFIRMADO

Dentro de `{{ range .ai_tools.curl }}` o `.` (dot) passa a ser **o elemento atual** do range
(o map da tool). Para alcançar dados na raiz usa-se `$`, que no Go template é sempre o
**contexto inicial** com que o template foi executado.

O script já comprova esse padrão: dentro do range ele lê o campo da tool com `.feature_flag`
e o estado global com `$.features.ai.coding_assistants` (linhas 12-13, 29-30). Portanto:

- Campo da entrada: `.block_on_workplace`
- Valor global do workplace: `$.settings.workplace`

`$.settings.workplace` é o caminho correto. Os templates `workplace/script_is_zup` /
`script_is_not_zup` leem `.settings.workplace` porque ali `.` JÁ é a raiz (são invocados com
`{{ template "..." . }}` no topo). Dentro do range precisamos do `$` para o mesmo dado.

## 2. Entradas SEM o campo — `hasKey` (sprig) + short-circuit

`hasKey` é função do **sprig** (`hasKey $map $key`) e retorna `true`/`false` se a chave existe
no dict. Já é o padrão do script para campos opcionais: `hasKey . "env"` (l.17, 34) e
`hasKey . "completions_cmd"` (l.19). A maioria das tools NÃO terá `block_on_workplace`, então a
checagem precisa do short-circuit de `and` (Go template avalia `and` da esquerda p/ direita e
só compara o segundo termo se o primeiro for truthy):

```gotemplate
(and (hasKey . "block_on_workplace") (eq .block_on_workplace $.settings.workplace))
```

→ Para entradas sem o campo, `hasKey` é `false` e `eq .block_on_workplace ...` nunca é avaliado
(sem erro de chave inexistente). Resultado: `false` = não bloqueado. ✅

## 3. Compor feature flag + guard de workplace — idioma recomendado

Em vez de aninhar tudo num `and (not (and ...))` ilegível, calcule uma variável `$blocked`
logo após o `$featureEnabled` (mesmo estilo das linhas 11-13) e combine com `and` no `if`.
Mais limpo, testável e consistente:

```gotemplate
{{- $blocked := false -}}
{{- if and (hasKey . "block_on_workplace") (eq .block_on_workplace $.settings.workplace) }}{{ $blocked = true }}{{- end -}}

{{ if and $featureEnabled (not $blocked) }}
```

Alternativa inline (1 linha, válida mas menos legível):
```gotemplate
{{ if and $featureEnabled (not (and (hasKey . "block_on_workplace") (eq .block_on_workplace $.settings.workplace))) }}
```
Recomendação: **variável `$blocked`** — combina com o estilo já existente do arquivo.

## 4. Pitfalls de whitespace / `{{-` no bash gerado

- O script mistura `{{- ... -}}` (linhas de lógica, sem emitir nada) com `{{ ... }}` (linhas que
  emitem bash). Mantenha as **declarações de variável** (`$blocked`) com hífens nos dois lados
  (`{{- ... -}}`) para não injetar linhas em branco no script bash final.
- A linha `{{ if and $featureEnabled (not $blocked) }}` deve seguir o padrão da linha 15/32
  **sem** hífen à esquerda (preserva a quebra de linha que separa o bloco anterior). Não altere
  os `{{- end }}` existentes (l.23-24, 35-37) — eles já balanceiam o whitespace do range.
- Insira o cálculo de `$blocked` **dentro** do range, ANTES do `{{ if }}`, espelhando onde
  `$featureEnabled` é montado. Não mova para fora do range (perderia o `.` da tool).
- Cuidado para não deixar `eq .block_on_workplace $.settings.workplace` comparar contra um
  campo que seria `<no value>` — o `hasKey` antes do `eq` (short-circuit) já previne isso.

## 5. YAML — adicionar o campo na entrada `bobibm`

```yaml
    - name: bobibm
      url: "https://bob.ibm.com/download/bobshell.sh"
      shell: bash
      feature_flag: coding_assistants
      block_on_workplace: Zup
```

Valor `Zup` casa exatamente com `$.settings.workplace` quando perfil = work Zup
(`ternary "Zup" "none" ...` em `home/.chezmoi.yaml.tmpl:241`). Demais workplaces (`none`,
work não-Zup) → `eq` falso → instala normalmente. ✅
